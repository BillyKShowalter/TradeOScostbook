# Brand Audit — Phase 1
*404 TradeOS (parent) + TradeOS (product). Recommendations only — nothing redesigned yet.*

## What exists today
- **404 TradeOS marketing site** — forge-black + copper, Space Grotesk, terminal/CNC motifs (crosshair, reticle cursor, CRT hover, scanlines). Confident, distinctive, memorable.
- **TradeOS CostBook** (product) — default shadcn neutral gray. Functional, generic, shares nothing visually with the parent brand.
- **Design system (this project)** — tokens, 11 components, 2 kits, 18 specimen cards, a pitch-deck template in progress, and (added this session) an enterprise elevation layer: `.theme-blueprint` (cool graphite/paper, for product) and `.theme-carbon` (cool dark alt), elevation/glass/motion tokens, a premium mono stack, tabular numerals.

## Strengths — preserve these
1. **Copper as sole accent** — disciplined, ownable, already correctly used everywhere. Do not add competing brand colors.
2. **Space Grotesk for display** — geometric, precise, reads as "engineered." Right typeface for both brands.
3. **The 404 badge lockup + wordmark** — distinctive, already documented, real SVGs on file. Don't touch.
4. **CNC/crosshair/measurement motifs** — a genuine differentiator ("industrial precision") not seen in competitor construction-tech marketing. Worth keeping as *marketing-only* signature, used tastefully.
5. **Voice** — direct, trade-literate, no hype. Already well-specified in readme.md with real examples. This scales fine to an enterprise platform without changes.

## Inconsistencies — the core problem
1. **No shared DNA between 404 TradeOS and TradeOS today.** The product (CostBook) looks like an unrelated app. A user moving from the marketing site into the product would see no family resemblance at all.
2. **One visual register, two audiences.** The forge-black/terminal treatment is right for *marketing* (bold, differentiating) but wrong for *dense daily-use software* (fatigue, low information density, novelty over 8 hours/day). Using it as-is for CRM/estimating/field-ops would violate "usability over visual effects."
3. **Mono stack was Courier New** — dated, undermines "premium engineering" positioning. *(Already refined this session to a premium system-mono stack; flagged here for the record.)*
4. **No light surface existed anywhere** — enterprise B2B software is overwhelmingly light-mode-first for dense data work. This was a genuine gap, not a stylistic choice. *(Also addressed this session via `.theme-blueprint`.)*

## Weaknesses — open questions, not yet solved
1. **No defined relationship rule** between 404 TradeOS and TradeOS lockups (parent/product logo lockup, when each appears, co-branding on the product's own marketing pages). Needs Phase 2.
2. **No component-level product patterns yet** — Table, Sidebar nav, data density rules, form patterns for a CRM/estimating tool don't exist. Needed before any product screens (Phase 6).
3. **No print/merch system** — business cards, letterhead, decks beyond the one pitch-deck template don't exist yet (Phase 7–8).
4. **Photography/illustration style undocumented** — real jobsite photography exists and works (verified), but there's no written rule for a "Construction Intelligence Platform" positioning (does imagery stay blue-collar-authentic, or shift toward abstract data/diagram illustration for enterprise credibility?). Needs a decision in Phase 2/3, not a guess.

## Recommendation for Phase 2 (not yet started)
Define the parent/product architecture explicitly: 404 TradeOS keeps the Forge (dark, expressive, motif-rich) register for marketing; TradeOS product adopts the new Blueprint (light, quiet, data-dense) register for daily-use software, sharing copper + Space Grotesk + voice as the unifying DNA. Both inherit one token source so they can never drift apart silently.
