## Watch Outs
- Never run tenant Prisma queries outside an HTTP, admin UI, or background database session once RLS is active.
- Never use PostgreSQL superuser credentials for application traffic or RLS tests.
- Deploy request-session code before or atomically with forced-RLS migrations.
- Keep `PLATFORM_PROVISIONING_SECRET` separate from JWT secrets, at least 32 characters, rotated, and restricted at the network edge.
- `PLATFORM_PROVISIONING_ALLOWED_IPS` relies on accurate `req.ip` — it only does anything useful if the app trusts a correctly configured proxy in front of it (Express `trust proxy`). Don't treat it as sufficient on its own; it's a backstop behind real network controls.
- Keep audit tables append-only; no update/delete policies exist by design.
- `supplier_price_updates` review (approve/reject) is restricted to admin/owner under RLS even though estimator+ can enqueue — this is deliberately tighter than ordinary tenant writes, since approval is what actually moves a live price. Don't loosen the review policy to match the enqueue policy without re-checking that intent.
- `supplier_price_updates.supplier_id`/`material_id` are `ON DELETE RESTRICT`, not `CASCADE` — deleting a supplier or material with any price-update history (including resolved rows) is supposed to fail, the same way `material_price_audits` protects materials. If you ever see this as `CASCADE` again, it's a regression, not a feature.
- The supplier price sync scheduler deliberately does not auto-discover organizations/suppliers — `SUPPLIER_PRICE_SYNC_JOBS` lists targets explicitly. Don't "simplify" this into a cross-tenant `organization.findMany()` scan using the app's normal restricted DB connection — under forced RLS with no `app.org_id` session set, that silently returns zero rows rather than erroring, which is exactly the kind of footgun this app's existing Watch Outs already warn about for tenant Prisma queries outside a request/background session.
- `prisma/migrations/` is the only authoritative migration history now — `db/migrations/` no longer exists. Never hand-edit an already-applied migration's `migration.sql`; add a new migration instead, the same as any other Prisma Migrate project.
- `DATABASE_ADMIN_URL` is for `scripts/deploy-migrations.sh`/`scripts/provision-app-role.sh` only — never the running API's `DATABASE_URL`. The restricted role can't run DDL or grant itself privileges, by design; that's exactly why these need a separate elevated connection.
- If you add a migration that introduces a new table needing the restricted role's access, you don't need to hand-edit grants — `scripts/sql/provision-app-role.sql` already sets `alter default privileges` for future tables/sequences/functions, and re-running `scripts/deploy-migrations.sh` (which always re-runs the grant statements too) catches anything that predates that default-privileges setting.
- `psql` must be on `PATH` for `scripts/deploy-migrations.sh`/`scripts/provision-app-role.sh` to run (host client, not `docker exec`-based) — on macOS via Homebrew it's keg-only (`brew install libpq`, not symlinked into `/opt/homebrew/bin` automatically).
- `.github/workflows/deploy-migrations.yml` has only ever been run via `act` against this repo's local disposable Postgres container, never against real GitHub infrastructure (no git remote exists yet) and never against a real production database. Treat it as unverified-in-the-target-environment until someone actually triggers it for real with the `production` Environment's reviewer gate configured — don't assume the local `act` pass means it's safe to point at a real database unattended.
- `/admin`, `/admin/pricing-history`, and `/admin/member-history` all now share one CSS file (`adminShellCss` in `api/views/adminShell.view.ts`). Changing a shared class (`.chip`, `.pagination`, `.access-panel form`, `.filters`, etc.) affects all three pages — check all of them, not just the one you're editing, before assuming a CSS tweak is page-local.
- Never run `prisma migrate dev` (or trust `prisma migrate diff` output) blindly against this schema without reading the diff first — because the two original migrations were hand-written SQL rather than Prisma-generated, Prisma's own diff engine sees the whole schema as drifted (wants to rewrite primary keys, change `uuid` columns to `TEXT`, rename every constraint/index) even when nothing real changed. This is cosmetic, not a bug to fix — write new migrations by hand (a fresh folder under `prisma/migrations/` + `migration.sql`), the same way the first two were written, and the same way `20260624083000_add_assembly_is_template` was added.
- Template assemblies (`isTemplate`) are org-scoped, not a shared cross-tenant catalog — a `NULL` `org_id` row would be invisible to every tenant under forced RLS regardless of intent. Don't try to add a "global" template library by setting `org_id` to `NULL`; it would silently show up nowhere.
- **Never ask the user to paste a raw database password/connection string into chat, and if one appears anyway, treat it as compromised immediately** — have them rotate it via the dashboard before using anything derived from it. This happened twice in one session despite repeated requests not to. Prefer: web UI for secret entry, or a local temp file piped into `gh secret set` (`cat > /tmp/x.txt`, then `gh secret set ... < /tmp/x.txt`, then `rm /tmp/x.txt`) over multi-line interactive stdin prompts or long single-line shell commands with the secret inlined — both of the latter are exactly what produced the leading-space corruption and the line-wrap corruption that wasted several round trips this session.
- Supabase's direct-connection host (`db.<project-ref>.supabase.co:5432`) is IPv6-only unless the project has paid for the IPv4 add-on. Any IPv4-only environment (GitHub Actions hosted runners, and likely most serverless platforms) must use the Session-mode Supavisor pooler instead (`aws-<N>-<region>.pooler.supabase.com:5432`, username `postgres.<project-ref>`) — Transaction mode (port 6543) doesn't support prepared statements/DDL and won't work for migrations. The pooler node number (`aws-0` vs `aws-1` etc.) isn't predictable from the project ref/region alone — read it off the actual dashboard connection string rather than guessing.
- The `production` GitHub Environment currently has no required-reviewer gate (the protection rule wouldn't enable via API even on GitHub Pro) — `workflow_dispatch` runs against it execute immediately with no approval checkpoint. Don't treat this environment as "protected" until that's resolved.
- Keep long-running work outside HTTP transactions and comfortably below `RLS_TRANSACTION_TIMEOUT_MS`.
