## 2. Naming Conventions

Consistent throughout: models are `PascalCase` singular, every field maps to `snake_case` via `@map`, every model maps to a `snake_case` plural table via `@@map`. No exceptions found. This is good hygiene that keeps the Postgres-native surface (which is what RLS policies, `psql`, and Supabase's dashboard all show) readable independent of the Prisma client surface.

One minor inconsistency: `OrganizationMembership` fields are declared with no blank-line separation from their relations in some models and with separation in others (e.g. `Organization` model has relation fields packed together with no grouping comments, while `Assembly` groups scalar fields, then a documented `isTemplate` field, then relations). Purely cosmetic â€” not worth a migration, just worth keeping in mind if this schema file is ever auto-formatted, since `prisma format` will not reorder fields and won't fix this on its own.

This is good discipline; it means the `mapPrismaKnownRequestError` P2003 handling in `backend/middleware/errorHandler.ts` (documented in `CLAUDE.md` item 30) has a fully-specified, intentional set of FK behaviors to work against rather than accidental defaults.

## 4. Field-Level Type Review

- **Decimal precision/scale choices are consistent and appropriate** for a construction cost book: `numeric(12,4)` for unit costs and quantities (cents-of-a-cent precision needed for e.g. per-square-foot pricing multiplied by large quantities), `numeric(14,2)` for rolled-up totals (currency-cent precision, larger range for project totals), `numeric(5,2)` for percentages, `numeric(6,4)` for regional index multipliers. No mismatched precision found â€” e.g. no case where a value computed from `numeric(12,4)` inputs is stored in a column with less precision than the computation could produce.
- **All PKs are `String @id @default(uuid())`**, mapping to Postgres `uuid` with `gen_random_uuid()` (via the `pgcrypto` extension enabled in the first migration) â€” consistent throughout, no mixed PK strategies (e.g. no stray `serial`/`autoincrement` model). Good for a multi-tenant system where IDs may need to be globally unique across orgs (avoids enumeration and merge-conflict-on-sequence issues in a distributed/branch-heavy Supabase workflow).
- **Nullable-vs-required judgment is generally sound.** `orgId` is nullable (`String?`) on most catalog tables (`Division`, `Region`, `Supplier`, `Material`, etc.) â€” worth double-checking this is intentional: a `NULL` `org_id` row is **invisible under forced RLS** to every tenant (confirmed correct understanding already documented in the `Assembly.isTemplate` migration comment: "`org_id = current_app_org_id()` is never true when org_id is NULL"). This means any pre-existing `NULL`-org catalog rows (if any exist from early seed/dev data) are permanently unreachable through the application today. Worth a one-time live check (`select count(*) from materials where org_id is null`, etc., against the real Supabase project) to confirm no orphaned rows exist â€” this audit did not have live database access and could not verify this directly.
- **`String` used for enum-like fields throughout** (`role`, `status` on many models) rather than a Prisma `enum`. This is a deliberate, defensible choice given the existing pattern (`CLAUDE.md` shows status values evolving over time â€” e.g. `invoices.status` gained `overdue` in a later migration than initial design) â€” Prisma enums generate a Postgres native `enum` type, which is more rigid to extend (`ALTER TYPE ... ADD VALUE` has real limitations, including not being usable inside a transaction in older Postgres versions) than a `text` column with an app-level or `check`-constraint-level enum. Given this schema already migrates status vocabularies over time, staying with `text` (+ `check` constraints where present, see master plan Â§5) is the right call for this codebase, not a gap.

## 5. Prisma Client Usage Patterns (from `db/client.ts`, `db/requestSession.ts`, and `modules/*/service.ts`)

- **Single shared Prisma client instance**, correctly guarded against hot-reload duplication (`globalForPrisma` pattern) â€” standard, correct.
- **Custom `Proxy` wrapper routes every model call through `AsyncLocalStorage`-scoped request transactions when active.** This is an unusual but well-reasoned pattern specific to this codebase's RLS design (every model method call transparently uses the current request's transaction client if one exists, falling back to the base client otherwise) â€” confirmed this is applied uniformly; no service file was found bypassing the proxy by importing `PrismaClient` directly instead of the shared `prisma` export. This uniformity is exactly what makes forced RLS safe here â€” a single service that imported `@prisma/client` directly instead of `db/client.ts`'s `prisma` export would silently run outside the RLS-scoped transaction and either fail (forced RLS blocking an unscoped session) or, worse, succeed against the wrong scope if run as a superuser role. Worth keeping as a standing lint/review rule for any new module.
- **No raw SQL used for actual data queries anywhere in the codebase.** Every `$queryRaw` call site (`db/requestSession.ts`, `backend/auth/session.ts`, `modules/auth/service.ts`, `modules/organization-provisioning/service.ts` â€” 10 call sites total) is the same intentional pattern: a `set_config('app.user_id'|'app.org_id'|'app.role'|'app.auth_subject'|'app.login_lookup', ..., true)` bootstrap call that makes the RLS session-variable model work, using `Prisma.sql` tagged templates throughout (correctly parameterized, no string concatenation / SQL injection surface). No `$executeRaw` usage at all. This is a clean, disciplined codebase in this respect â€” every actual data read/write goes through Prisma's query builder, which keeps it inside the request-transaction proxy automatically; raw SQL is reserved exclusively for the one thing Prisma's query builder can't express (session-local `SET`).
- **No `include`/`select` over-fetching found** in the models reviewed â€” most list/search queries return whole rows (appropriate given DTOs generally mirror the table shape) and `include` is used narrowly (e.g. `costItem.findFirst({ include: { laborRate: { include: { region: true } }, material: true, equipment: true } })` in `cost-database/service.ts:132` pulls exactly the three optional cost components a cost item can reference, nothing extraneous).
- **One N+1 query pattern**, detailed fully in `docs/DATABASE_MASTER_PLAN.md` Â§8.3: `AssembliesDatabaseService.getAssemblyUnitCost` resolves each assembly line item's cost with a separate `CostDatabaseService.getUnitCost()` call inside a loop, rather than batching. Not repeated in full here â€” see the master plan for the concrete fix recommendation.
- **Sequential-numbering pattern** (`aggregate({ _max: ... })` then `create` as two separate statements, in both `change-orders/service.ts` and `invoices/service.ts`) is a genuine minor race condition, also detailed in the master plan Â§8.2 rather than duplicated here.

## 6. Search Query Review (Prisma-specific angle)

Every `contains`-based search in this codebase passes `mode: "insensitive"` â€” correct, consistent use of Prisma's case-insensitive filter mode (which Postgres implements as `ILIKE` under the hood). The gap is not correctness, it's the missing supporting index (full detail in master plan Â§4) â€” `mode: "insensitive"` plus `contains` together compile to a query shape (`ILIKE '%...%'`) that a plain btree index (even a case-insensitive one) cannot accelerate; only a trigram GIN index can. Flagging here specifically because it's easy to assume "the field is indexed" (e.g. `code` has a unique index via `@@unique([orgId, code])`) without realizing that index is useless for the actual query Prisma is generating.

## 7. Recommendations Specific to Prisma Workflow

1. **Migration authoring convention is already well-established and should continue:** hand-write SQL migrations rather than trusting `prisma migrate diff`/`prisma migrate dev`'s auto-generated diff against this schema, per the documented lesson in `CLAUDE.md` item 34 (the auto-diff engine wanted to rewrite PK types and every constraint name across nearly the whole schema â€” cosmetic noise specific to this schema's hand-written-SQL lineage, not a real diff). Any of the migrations recommended in the master plan's Â§10 should follow this same hand-written pattern.
2. **Consider adding `prisma format` and `prisma validate` as explicit steps in CI** (or confirm they already run as part of `npm run build`/`npm run lint` â€” not confirmed in this read-only audit) to catch schema/migration drift automatically going forward, rather than relying on manual review like this one.
3. **No `@@map`/`@map` issues, no missing `@updatedAt`, no missing `@default(now())` on any `createdAt`** â€” checked every model for these common Prisma oversights; none found. This schema does not need a "fix the basics" pass.
4. **`Prisma.Decimal` vs `Number` boundary** â€” see master plan Â§8.5. Not a Prisma issue per se (Prisma correctly returns `Prisma.Decimal` objects for every `Decimal`-typed column), but worth noting at the Prisma-review level that the *entire codebase* converts every `Decimal` to `Number` at the DTO boundary via a consistent `toXDTO()` mapper per module â€” this is the right pattern already; the risk is only in new modules that forget to route a Decimal field through the mapper (exactly the bug class already caught twice per `CLAUDE.md` item 40).

## Summary

This is a clean, disciplined Prisma setup: no schema/migration drift, no missing conventions, thoughtful `onDelete` choices throughout, and consistent, uniform use of the request-transaction proxy that makes forced RLS actually safe.
